Relatório de Integração Tecnológica: API do CE Mercante e Conectividade SiscargaArquitetura de Integração Aduaneira e Evolução dos Sistemas de CargaO cenário do comércio externo brasileiro passou por transformações que visaram desburocratizar e unificar o controlo aduaneiro de mercadorias. A transição das antigas plataformas de troca de ficheiros e bases de dados legadas para interfaces modernas de programação de aplicações (APIs) representou um salto qualitativo para importadores, exportadores e recintos alfandegados. O ecossistema do Sistema Mercante e do Sistema de Controlo de Carga Marítima (Siscarga) encontra-se atualmente estruturado sob o barramento de serviços do Integra Comex, mantido pelo Serviço Federal de Processamento de Dados (SERPRO).A evolução tecnológica das APIs aduaneiras baseia-se na arquitetura REST (Representational State Transfer), que adota o formato JSON para intercâmbio de dados e a codificação UTF-8 para assegurar a compatibilidade de caracteres especiais. Um marco recente desse processo foi a publicação da Notícia Siscomex Sistemas nº 013/2024, que introduziu a API do Controle de Carga e Trânsito (CCT) Importação para o Modal Aquaviário, desenhada para interagir com a Declaração Única de Importação (Duimp) e com o Conhecimento de Embarque (CE) Mercante.A base de dados ativa do sistema de carga marítima abrange dados registados a partir do ano de 2017. Operações efetuadas em períodos anteriores a 2017 devolvem um payload vazio na chamada da API, exceto se os registos tiverem sofrido alguma modificação posterior a essa data, situação na qual é retornada apenas a data e a hora da respetiva modificação.Procedimentos de Conexão e Autenticação no Integra ComexA conexão com as APIs do SERPRO utiliza o protocolo OAuth2 e exige um duplo fator de validação na fase operacional. O fluxo de segurança é dividido entre a parametrização contratual e a execução operacional de chamadas.Fase Contratual e Obtenção de CredenciaisO acesso às bases restritas — as quais contêm dados protegidos por sigilo comercial e fiscal — requer a contratação prévia do serviço Integra Comex através da Loja Serpro. A validação do contrato é efetuada mediante o uso de um certificado digital e-CNPJ da empresa contratante, emitido segundo o padrão da Infraestrutura de Chaves Públicas Brasileira (ICP-Brasil). Após a contratação, o cliente obtém acesso à sua Área do Cliente Serpro, onde são geradas as seguintes credenciais de acesso:Consumer Key: Identificador público que associa o sistema integrador ao contrato assinado com o SERPRO.Consumer Secret: Chave criptográfica privada utilizada para assinar os pedidos de geração de tokens.Fase Operacional e Geração do Token de Acesso (Bearer)Para efetuar consultas ou transmissões nos endpoints do CE Mercante e do Siscarga, a aplicação do utilizador deve realizar um handshake SSL apresentando um certificado digital e-CPF (padrão ICP-Brasil, do tipo A1 ou A3) associado ao representante legal ou operador autorizado.O processo para a obtenção do token temporário de acesso (Bearer Token) consiste em enviar uma requisição HTTP do tipo POST para o endpoint de autenticação do SERPRO.As URLs de acesso para o processo de autenticação variam conforme o ambiente técnico:AmbienteURL do Endpoint de Autenticação (SAPI)Validação / Homologaçãohttps://hom-siscomex-sapi.estaleiro.serpro.gov.br/authenticate[cite: 2]Produçãohttps://siscomex-sapi.estaleiro.serpro.gov.br/authenticate[cite: 2]Para realizar a requisição, o sistema do utilizador deve estruturar os cabeçalhos HTTP (Headers) com as seguintes propriedades:Nome do CabeçalhoTipoValor / DescriçãoAuthorizationStringPrefixo Basic  seguido da string Consumer Key:Consumer Secret codificada em Base64.role-typeStringPerfil do utilizador. Endpoints e Comandos da API do Siscarga Marítima (Consultas Restritas)Os endpoints de consulta restrita exigem que as chamadas HTTP enviem dois tokens de validação nos cabeçalhos de cada requisição: o Authorization contendo o valor Bearer <access_token> e o jwt_token contendo o token JSON Web Token recuperado no ato da autenticação.As URLs de base para o consumo das APIs do Siscarga Marítima variam de acordo com o ambiente operacional:Ambiente de Validação (Homologação): https://gateway.apiserpro.serpro.gov.br/integra-comex-carga-hom/v1[cite: 2]Ambiente de Produção: https://gateway.apiserpro.serpro.gov.br/integra-comex/api[cite: 2]As tabelas de comandos da API do Siscarga estão estruturadas por tipo de operação aduaneira.Consultas de Conhecimento de Embarque (CE Mercante)Método HTTPURI do EndpointParâmetros de PathFunção e Regras de NegócioGET/conhecimentos-embarque/{nr}nr: Número do CE Mercante.Retorna os dados gerais (capa do documento) de um conhecimento de embarque específico. O número do CE não é validado sintaticamente, sendo retornados apenas os registos exatos encontrados na base.GET/conhecimentos-embarque/{nr-ce}/itens/nr-ce: Número do CE Mercante.Permite a consulta detalhada de todos os itens associados ao CE Mercante informado. Caso o volume de dados ultrapasse o limite de retorno, a API fornece no payload atributos links com URIs para paginação (rel="next" e rel="self").GET/conhecimentos-embarque/{nr-ce}/itens/{nr-item}nr-ce: Número do CE Mercante;nr-item: Número do item inicial de pesquisa.Consulta os itens do CE Mercante a partir de um índice numérico de início (nr-item). Se este parâmetro for omitido, a API assume o primeiro item físico existente no documento como ponto de partida da resposta.Consultas de Escala e Manifesto MarítimoMétodo HTTPURI do EndpointParâmetros de PathFunção e Regras de NegócioGET/escalas/{nr}nr: Número da Escala (formato de 11 dígitos, 'NNNNNNNNNNN').Retorna o detalhamento cadastral e logístico da escala do navio. Utilizado para confirmar a consolidação de cargas do navio.Endpoints de Monitorização e Atualizações PúblicasPara otimizar o consumo das APIs e evitar o processamento desnecessário de dados, o SERPRO disponibiliza endpoints com visibilidade pública irrestrita. Estas chamadas devolvem dados sobre a data e hora da última modificação sofrida pelo registo aduaneiro.Desta forma, os ERPs privados podem implementar rotinas de varredura (polling) eficientes: o sistema efetua uma chamada leve a estes endpoints públicos e, caso o carimbo de data/hora retornado seja superior ao registado localmente, dispara a consulta restrita autenticada correspondente para extrair o payload completo.As consultas públicas usam os seguintes endpoints da API:Método HTTPURI do Endpoint PúblicoTipo de Retorno esperadoObjetivo do ComandoGET/ce/data-ultima-atualizacaoData/Hora da alteração.Verificar se houve atualizações nas informações declaradas no Conhecimento de Embarque Mercante.GET/escala/data-ultima-atualizacaoData/Hora da alteração.Validar modificações no registo de escala da embarcação transportadora.GET/manifesto/data-ultima-atualizacaoData/Hora da alteração.Monitorizar alterações ou inclusões de novos conhecimentos na estrutura de manifestos consolidados.Endpoints do Controle de Carga e Trânsito (CCT Importação - Modal Aquaviário)A integração com as novas diretrizes do Portal Único Siscomex exige que os operadores privados interajam diretamente com o módulo CCT Importação para cargas do modal aquaviário. recintos aduaneiros.Regras de Validação Criptográfica e Restrições de NegócioPara além do protocolo de autenticação e da chamada de endpoints, a integração entre os sistemas corporativos e o barramento governamental do SERPRO exige a conformidade com regras estritas de infraestrutura aduaneira, concebidas para mitigar erros de integridade de dados e prevenir fraudes fiscais.Sincronização de Relógios e Validação TemporalDevido à necessidade de garantir a irretratabilidade das transações, a assinatura criptográfica assente no padrão JWT impõe uma verificação em tempo real da validade do carimbo de data/hora (timestamp) enviado na chamada de autenticação. Os servidores privados envolvidos no envio das mensagens aduaneiras devem possuir os seus relógios internos estritamente sincronizados com o protocolo de tempo padrão brasileiro (NTP.BR). Qualquer discrepância temporal que exceda o limite de tolerância técnica resulta na rejeição imediata da autenticação, devolvendo erro de expiração de token ou assinatura inválida.Sanitização e Regras de Formatação de IdentificadoresA validação de máscara de campos no gateway aduaneiro proíbe o envio de caracteres especiais e de formatação textual para dados sensíveis.Identificadores Fiscais (CPF e CNPJ): Devem ser transmitidos exclusivamente sem máscara (apenas dígitos numéricos), sendo rejeitado o uso de pontos, hifens ou barras.Conhecimento de Carga: Deve ser submetido sem formatação de pontos, admitindo-se apenas a inclusão de hifens na sua composição alfanumérica.Declarações Aduaneiras: Campos contendo números de declarações (DI, DTA, Duimp, DUE, DAT) devem ser enviados estritamente sem máscara, respeitando-se as regras de tamanhos exatos exigidos por cada sistema governamental específico.Limite Físico de Transmissão (Payload)A API de integração estabelece um limite máximo de 500 KB por pacote de dados enviado. As requisições que excedam esta dimensão de payload são sumariamente rejeitadas com erro de validação de tamanho de pacote, obrigando a que os dados de lotes volumosos sejam fracionados e enviados de forma paginada ou assíncrona.Políticas de Idempotência e Prevenção de DuplicadosNa transmissão de eventos aduaneiros (especialmente no módulo de eventos de recintos alfandegados e operações do CCT), a API utiliza o campo idEvento como identificador exclusivo de transação. Para evitar o processamento duplicado de mensagens causado por falhas de conectividade ou reenvios intempestivos, o barramento adota as seguintes regras de negócio:Se o recinto enviar uma mensagem com um idEvento que já tenha sido processado pelo sistema aduaneiro nos últimos 60 minutos (considerando o mesmo CNPJ e tipo de evento), a API rejeita a inclusão e recupera o protocolo de envio original correspondente ao primeiro processamento.Esta validação garante que oscilações na rede de internet privada das instalações portuárias não gerem duplicidade de registos físicos ou faturas duplicadas.Bloqueio Aduaneiro e Sanções PecuniáriasA amarração física e fiscal do CE Mercante com o fluxo de desembaraço da Duimp estabelece uma dinâmica operacional rígida. O agente de carga ou transportador deve declarar a Nomenclatura Comum do Mercosul (NCM) de forma precisa e sem divergências. No momento em que o importador submete o registo da Duimp, o Conhecimento de Embarque (CE) correspondente é automaticamente bloqueado no sistema aduaneiro contra retificações diretas na base.Caso seja constatada alguma divergência de informações pós-registo aduaneiro, qualquer retificação depende da instauração de um procedimento administrativo fiscal. Essa correção fora do prazo legal ou após a atracação do navio dispara de forma automática multas administrativas no valor de R$ 5.000,00 por ocorrência (com base no art. 107, IV, "e" do Decreto-Lei nº 37/1966), além de bloquear a parametrização da carga, gerando atrasos aduaneiros graves e custos adicionais com sobre-estadia de contentores (demurrage).Conclusões e Diretrizes de Implementação TécnicaA conexão com as APIs do CE Mercante e Siscarga Marítima requer o cumprimento de elevados padrões de segurança criptográfica e conformidade técnica aduaneira. As plataformas de integração das empresas privadas devem ser estruturadas de forma robusta e resiliente.Para garantir a continuidade operacional das integrações de sistemas de comércio externo, recomendam-se as seguintes diretrizes arquiteturais:Tratamento de Exceções de Media Type: O sistema de processamento de filas deve identificar erros de protocolo 415 Unsupported Media Type nas chamadas ao gateway de tokens, ajustando dinamicamente o cabeçalho Content-Type para o formato application/x-www-form-urlencoded com o parâmetro grant_type=client_credentials.Sincronização Contínua de Tempo (NTP): Configurar um serviço daemon ativo de sincronização de relógio de hardware com o servidor público gps.ntp.br nos servidores de aplicação do ERP, de forma a anular falhas de descompasso de tempo na geração das assinaturas JWT.Filtragem Preventiva Prévia ao Registo: Implementar um passo de validação lógica (checklist) no sistema interno que consume as APIs de capatazia e frete do CE Mercante antes de iniciar a transmissão dos dados para registo ou retificação da Duimp. Isto previne a ocorrência de divergências de NCM e anula a aplicação de sanções pecuniárias decorrentes de bloqueios indesejados pós-submissão aduaneira.Processamento Orientado a Eventos com Idempotência: Armazenar sistematicamente as chaves de idEvento geradas localmente pelo transmissor. Na ocorrência de timeout de conectividade, a aplicação integradora deve primeiro tentar recuperar o estado do protocolo através do identificador único original, impedindo o envio de comandos idênticos dentro da janela de carência de 60 minutos estipulada pela Receita Federal.
